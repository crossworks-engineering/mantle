# v0.150.0 — Maintenance runner (registry + `pnpm maintain` CLI)

Phase 1 of the maintenance-runner plan (docs/maintenance-runner.md), following
a full audit of the ~22 operational scripts.

## What changed

- **Maintenance task registry** (`apps/web/lib/maintenance/registry.ts`) — one
  source of truth describing every ops/backfill script: kind
  (recurring/remedy/ops/backfill), status (live/retired), cost
  (sql/io/imap/crypto/embedding/llm), flag convention, required env, and
  whether it's eligible for the future cron worker. A load-time assertion plus
  `registry.test.ts` enforce the cost-safety guardrail: model-spending tasks
  can never be marked schedulable.
- **`pnpm maintain`** (`apps/web/scripts/maintain.ts`) — a general terminal
  runner over the registry: `list [--all]`, `info <slug>`, and `<slug>
  [flags…]` which spawns the underlying script with flags passed through.
  Generic `--apply` translates to each script's own live flag (e.g.
  `entities-dedupe --go`). Safety rails: live runs of llm/embedding-cost tasks
  require `--yes`; retired backfills require `--force-retired`; missing
  required env fails fast.

## Audit conclusions (recorded in the doc)

- `dedupe:edges` does **not** need recurring runs — the extractor is
  delete-then-insert idempotent and the dashboard monitors the duplicate
  count; it's classified as a `remedy`.
- The only genuinely recurring hygiene job is `entities-dedupe`; backups
  already run on the `db-dump.sh` cadence.
- Heartbeats were rejected as the scheduling home (LLM-only, no cron);
  Phase 2 will use a pg-boss cron worker; Phase 3 adds a Maintenance tab to
  `/debug/integrity`.

No schema changes, no runtime behaviour changes — new CLI + docs + tests only.
