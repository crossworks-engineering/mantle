# 0.36.0 — 2026-06-23

One template, everywhere — onboarding now derives AI workers and the persona from
the manifest, closing the last split-brain in the config system.

## Highlights

- **AI workers are manifest-driven.** The worker models, params, and the
  OpenRouter→xAI voice routing used to live as hardcoded constants in the
  onboarding provisioner *and* (partially) in `MANIFEST_WORKERS`. Now the manifest
  is the single source: each worker declares its provider, model, params, and an
  optional dedicated alt route (voice upgrades to xAI when that key exists).
  Onboarding seeds workers through the shared `seedManifestWorkers`, so there's
  one place to change a worker model — and it can no longer drift from what the
  `/settings/config` checker measures against.

- **The persona is manifest-driven too (structure).** A fresh persona's model,
  parameters, context budgets, and tool groups now come from the manifest persona
  entry. The only things onboarding still owns are the genuine user *overlay*: the
  generated prompt (persona bank + the personality step), the assistant's name,
  and the voice that follows the chosen gender. The persona prompt remains
  operator-owned, exactly as before.

- **New required workers reach existing brains on upgrade.** The once-per-version
  boot reconcile now provisions any *missing required* worker from the manifest
  (gap-fill). It never overwrites an existing worker's model or provider —
  operator cost/model choices stand — and optional media workers are left to
  onboarding.

## Why this matters

A single app-builder fix typically spans a tool, a tool-group, a skill body, and
a specialist prompt — and every one of those already propagated to existing
brains from the manifest. Workers and the persona's base definition were the two
surfaces onboarding still kept its own copy of. With this release, **every config
surface the sanity system tracks has exactly one source of truth**, and onboarding
is `manifest + user overlay` end to end.

## Notes

- No schema migration in this release. Worker routing is unchanged behaviourally:
  a fresh brain seeds the same workers/models as before; voice still upgrades to
  xAI when an xAI key is present.
- The `/settings/config` checker treats a voice worker on its declared xAI alt
  route as in-sync (not drift).
