# 0.35.0 — 2026-06-23

Specialist prompt fixes now reach existing brains on upgrade — not just fresh installs.

## Highlights

- **Shipped specialist prompt/model fixes propagate on update.** The
  once-per-version boot reconcile now force-syncs each manifest specialist's
  product-owned definition — its system prompt, model, and parameters — to the
  template, the same way it already force-syncs manifest *skill* bodies. Until
  now a fix that lived in a specialist's **agent prompt** (for example the
  Appsmith *build → declare → call* ordering in 0.34.1 that stops an app calling
  an undeclared tool) only reached newly-onboarded brains; an already-running box
  kept the old prompt because gap-fill never overwrites an existing agent's
  prompt. Now it lands on the next version bump.

- **Scoped and safe.** Only manifest *specialists* are synced — the **persona is
  never touched** (its prompt is operator-owned). Tool groups, skills, and
  delegation are left to the additive grants, so capabilities you've *added* to a
  specialist survive; a specialist you've **disabled** is skipped (an opt-out, not
  reset). The model honours the same per-agent env override the seed uses.

## Why this matters

A single app-builder bug fix typically spans the whole config surface our sanity
system tracks — a tool, a tool-group, a skill body, and the specialist's own
prompt. Skills, tool groups, and brand-new specialists already reached existing
brains automatically; the agent prompt was the last piece that didn't. With this
release every part of a shipped fix reaches an existing brain on upgrade, and the
`/settings/config` checker (0.32.0) shows any remaining drift until it's applied.

## Notes

- No schema migration in this release.
- Applied on the next boot after `docker compose pull && up -d`. To force it
  sooner, or if boot reconcile is disabled, run the relevant
  `ALLOWED_USER_ID=<uuid> pnpm -C apps/web seed:<agent>`.
