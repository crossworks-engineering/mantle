# v0.153.1 — Maintenance phase-2 audit hardening

Fixes from two adversarial review passes (correctness/data +
concurrency/lifecycle/ops) over v0.153.0's cron worker + run history. No
high-severity findings; everything below was medium/low.

## What changed

- **Stop-request lifecycle in the UI run-store** — cancel/timeout no longer
  mark the run finished before the child actually exits: the single-flight
  lock stays held until `close` fires (so a SIGTERM-trapping script can't be
  overlapped by a second run), with SIGKILL escalation after a 10 s grace.
  Also collapses the duplicate/unordered history writes those paths could
  produce.
- **Cross-process merge lock** — applying `runEntitiesDedupe` (CLI, UI
  spawn, or nightly cron — three different processes) now takes
  `pg_try_advisory_xact_lock` on a slug-derived key; a contender fails fast
  with a clear message. Prevents two rare-but-real interleavings: alias
  lost-updates into the same canonical, and candidate-chain merges leaving
  edges pointing at a deleted entity (entity_edges has no FK). Dry-runs
  don't lock.
- **Stale-`running` reaping** — rows orphaned by a dead process (CLI Ctrl-C,
  container restart mid-run) flip to `failed` after 35 min, before history
  reads and at worker boot, instead of showing "running" forever.
- **Cron-side sweep timeout** — each sweep races a 30-min deadline (parity
  with the UI timeout) so a wedged merge can't hang the night's job with a
  perpetually-`running` row.
- **Worker robustness parity with email-sync** — SIGINT/SIGTERM graceful
  `boss.stop()`, `unhandledRejection` keep-alive backstop.
- **Cron timezone made explicit** — `boss.schedule(…, { tz: 'UTC' })`; the
  comment/docs claimed "server-local" but pg-boss defaults to UTC (03:30 UTC
  = 05:30 SAST; still off-peak).
- **CLI prints candidates before merging again** (parity with the
  pre-refactor script) via an `onCandidates` hook, so a mid-run failure
  still shows the detected list.
- **Toast context memoized** (pre-existing, app-wide) — a persistent API 5xx
  could loop fetch→toast→re-render→refetch on any screen whose load effect
  depends on `useToast()`; the context value is now stable.
- Stale comment in migration 0128 corrected (cron runs in the dedicated
  maintenance worker, not the events worker).

No schema changes; migration 0128 text change is comment-only (the runner
gates on the journal, not file hashes).
