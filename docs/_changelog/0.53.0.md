# 0.53.0 — 2026-06-24

Tables can now be counted, grouped, and aggregated in one call — and a windowed
read says out loud when it's incomplete.

## Highlights

- **`table_aggregate` — the GROUP BY tool.** `group_by` columns + optional
  `metrics`, `filters`, `match`, `sort` (`{by,dir}` over count / a group column /
  a metric), and limit/offset paging. Answers "count by metallurgy", "max design
  pressure by service", "distinct damage types" in one call instead of paging +
  grouping by hand. Built on a new pure `groupRows` in `table-model.ts`.

- **Filtered aggregates.** `table_query` takes `aggregate: [{column, kind}]`,
  computed over the **full matched set** (not the 500-row page). "Max design
  pressure among CS circuits" is one cap-immune call, no rows read back.

- **Truncation signal.** `table_query` / `table_get` / `table_rows_list` now
  emit `truncated` + `next_offset` + a `hint` whenever the returned slice is
  smaller than the true total. Every windowed read caps `rows` at 500 — but
  nothing in the payload said so, so an agent counting/maxing/listing over the
  returned rows would silently undercount on a >500-match filter (routine on the
  1k–3.3k-row tables in the field). Incompleteness is now self-announcing.

## Notes

- `table_aggregate` auto-joins the `tables` authoring group via
  `TABLE_TOOL_SLUGS`, so the Tables specialist (Ledger) gains it on existing
  brains via tool-group sync on image update.
- No schema migration in this release.
