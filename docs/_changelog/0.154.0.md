# v0.154.0 — Team-share toggle on every shareable type

Closes the sharing-parity gap on the owner side: the **"Team members only"**
admission toggle in the Share popover was offered on pages, apps, tables, and
folders — but not on **notes, tasks, events, or single files**, so those types
could only mint public links from the UI even though the backend has been
mode-agnostic all along (`shares.settings.mode`, enforced uniformly across the
whole `/s/` surface since v0.120.0).

There was no technical blocker — the toggle simply was never wired when team
mode shipped for pages/apps and later tables/folders. The `/team` member
workspace already lists shares of **both** modes in every section (with a
`public` badge on public ones), so team-sharing any of these types surfaces it
there immediately.

## What changed

- `<ShareControl teamMode>` is now passed on the four remaining shareable
  detail surfaces: task detail, event detail, note detail, and the single-file
  editor. No backend change — `PATCH /api/shares/[id] { mode }` was already
  type-agnostic.
- The default team-toggle hint no longer claims (stale) that "team-shared
  pages appear on the Team Hub"; it now says the item lists in the team
  workspace either way and the toggle only controls who can open the link.
- `docs/sharing.md` §7 documents the uniform toggle and the both-modes
  `/team` listing.

Journals, contacts, secrets, and emails remain unshareable by design
(rejected at the share API).
