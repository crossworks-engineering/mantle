# 0.34.1 — 2026-06-23

Mini apps no longer fail silently when they call a tool they never declared.

## Fixes

- **Undeclared tool calls are caught at build time.** A mini app can only call an
  api_tool that's (1) actually built and (2) declared in the app's tools via
  `app_tools_set` — the host refuses anything else. Until now a mismatch only
  surfaced as a runtime 403 the user hit when the call fired (e.g. *"tool
  'openweather_geocode' is not declared by this app"*). `app_build` now scans the
  source for every `host.tools.call('slug')` and **warns** (with file/line) for
  any slug that isn't declared, so Appsmith's build→fix loop and the Build panel
  see it immediately. Literal slugs only — dynamic ones are skipped to avoid
  false positives.

- **Friendlier runtime message.** If an app does reach a tool it hasn't declared,
  the broker's error now reads in plain language, and the /apps preview surfaces
  a clear toast naming the tool — even when the app's own code swallows the error.

## Notes

- **Appsmith prompt tightened.** The `app_authoring` skill and the Appsmith agent
  prompt now state the build→declare→call order explicitly, forbid inventing a
  tool slug, and tell it to delegate tool-building to the Toolsmith and treat the
  new build warning as must-fix. The skill body syncs on the next boot reconcile;
  the agent prompt requires `seed:appsmith` (mode overwrite) to reach an existing
  brain.
- No schema migration in this release.
