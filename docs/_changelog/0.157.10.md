# v0.157.10 — Runner queues slice 3 WP3: `ask_human` gates (still dark)

Third build package of the audited slice-3 plan
([docs/runs-slice-3-plan.md](../runs-slice-3-plan.md) §4 WP3 + its three
audit amendments): a run can now park on a human question. Table-engine work
— no DBOS involvement; the feature stays dark behind `MANTLE_RUNS`.

## Added

- **`ask_human` run items** (`{kind:'ask_human', question, options?,
  timeout_seconds?}`, seq-only): promote parks the item `ready` — never
  dispatched — and creates a `pending_tool_calls` row as the approval
  surface (existing /pending UI + `pending_*` tools + telegram flow; no new
  UI). The audit-item pattern with a human in the LLM's place.
- **The answer channel** (audit amendment 1): `pending_approve` gained an
  optional free-text `answer` (MCP tool + PATCH /api/pending/[id]); both
  approve and reject branch BEFORE tool resolution — there is no registered
  'ask_human' tool, by design — and complete the run item via the new
  `applyHumanAnswer` in `@mantle/runs`: `done` with `result.answer` (default
  `'approved'` for yes/option questions) or `failed({type:'rejected'})`, the
  counter driving the run forward either way. `pending_reject` on a question
  completes the item too — a rejected question left dangling would wedge its
  group forever.
- **Row lifecycle** (amendment 2): sweep duty 4 — the janitor — expires any
  pending question row whose item went terminal (run cancelled, branch
  fail_fast-cancelled, question timed out), and an answer landing on a dead
  item flips the row `expired` with a teaching error instead of pretending
  it took effect.
- **Dated questions** (amendment 3): `timeout_seconds` stamps the deadline
  at PROMOTE (like an audit's verdict budget — the item is never claimed)
  and sweep duty 1's predicate now includes ready `ask_human` items, so an
  expiring question actually expires (`failed(timeout)`, counter driven).
  Undated questions have NULL deadlines, never match duty 1, and are
  excluded from duty 2's re-dispatch (alongside audits) — they wait
  indefinitely, which is the feature.

## Changed

- `requires_confirm` items teach the working pattern now: gate the phase
  behind an `ask_human` step and run the gated tool inline at resume.
- `run_plan`'s description + plan schema document the new leaf; the parser
  teaches seq-only placement and requires a non-empty question.

## Notes

- Engine suite 22 → 27 (promote/pending-row shape, answer + seq advance,
  reject + duplicate-answer refusal, sweep exemptions + janitor after
  cancel, dated-question expiry) — all green against the workstation
  Postgres; `pnpm verify` green.
- The engine's pending-row insert leaves `agent_id` NULL on purpose:
  `runs.agent_id` is a soft ref that survives agent deletion, but
  `pending_tool_calls.agent_id` is a real FK — the args carry the run/item
  refs instead.
