# v0.148.1 — Search API hardening (audit follow-ups)

Fixes from the adversarial review of v0.148.0's `GET /api/search`.

## What changed

- **`tags` filter actually works now.** `searchNodes` bound the tags array as
  a raw JS array, which postgres-js serializes as a plain string — the
  `::text[]` cast threw and any `?tags=` request 500'd. Now routed through
  `pgArrayLiteral` like every other array filter (this also fixes the same
  latent path under the `search_nodes` MCP tool).
- **`branch` length caps.** Labels over 255 chars (Postgres's ltree limit) or
  branches over 500 chars are a clean 400 `invalid branch` instead of a cast
  error 500.
- **Strict-integer `limit`.** `limit=1e3` now clamps to 50 (parseInt read it
  as 1); fractional or suffixed values (`2.9`, `5abc`) are a 400.
- **Chunks mode carries `supersededBy`.** Passage hits from superseded nodes
  now name their living successor `{id, title, url}` — parity with nodes mode
  and the `search_chunks` MCP tool.

**Deploy: tag-only bump — no migration, no compose change.**
