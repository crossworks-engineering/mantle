# v0.158.1 — the run views repaint themselves

`/runs` was a photograph. The first real run on a live brain (the v0.158.0
dogfood, on a box with the page open the whole time) was created, parked on a
question, answered, resumed and completed while the list kept rendering **"No
runs yet."** Only a manual reload showed any of it.

That empty state is the worst version of the bug: it doesn't look stale, it
looks *settled*. An operator watching the page for a delegated job reads
"nothing is happening" at the exact moment the run is blocked waiting on them.

## Fixed

- **`/runs` is live.** The list, the selected run's item tree, and the
  active-runs strip on `/assistant` now repaint as a run is created, advances,
  parks, and finishes — no reload.
- **The active-runs strip can now appear at all.** It polled only while it
  already had a run (`refetchInterval` returning `false` on an empty list), so
  the one transition it exists to show — nothing → a run just started — was
  the one it could never make on its own.
- **The detail pane too.** It had no refresh of any kind; a tree opened
  mid-flight stayed frozen at whatever the first fetch returned.

## Added

- **`runs_changed` realtime channel** (migration `0135`). Triggers on `runs`
  and `run_items` raise `pg_notify('runs_changed', <owner id>)`; the web app's
  existing LISTEN bridge relays it to the browser as a `run` change, the same
  path `pending_changed` already uses for the approvals badge.

### Why a trigger, and not a notify in the engine

The engine mutates runs from several processes and paths — the `run_*` tools
in web and api, the three queue handlers, the minute sweep, the DBOS turn
workflows. A hand-placed notify is correct until the next path forgets one,
and this whole class of bug is *something didn't announce itself*. A trigger
cannot be bypassed by a code path that doesn't know about it.

It is also transactional, which is exactly the semantics wanted here: nothing
is delivered for a rolled-back engine transaction, and identical payloads
raised inside one transaction collapse, so a batch promoting ten items wakes
the browser once rather than ten times.

The `UPDATE` triggers carry `WHEN` clauses. Engine transactions write
bookkeeping columns (`attempt`, `deadline_at`, `usage`, cost) far more often
than anything these views render, and firing on those would be a notify storm
that repaints nothing.

### Why not polling

It was the cheaper fix and it is the wrong one here. `refetchOnWindowFocus` is
off app-wide, so a view that misses a change has no way back — which is why
the stale page stayed stale for minutes rather than self-healing on focus.

A poll does survive as a **safety net behind** the channel, because
LISTEN/NOTIFY has no replayable backlog and a change raised during a reconnect
gap is simply lost: 15s while a run can still move, 60s idle. The list's
backstop is deliberately *not* conditional on the list being non-empty — that
was precisely the strip's bug, and gating a poll on the state you are trying
to leave reproduces it.

## Not covered

The channel is best-effort, like every other realtime type here: it is a ping
to refetch, not a data channel. A browser that is disconnected at the instant
of the only change it cared about catches up on the backstop poll, not
instantly.
