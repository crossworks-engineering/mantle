# v0.157.8 — Runner queues slice 3 WP1: durable worker turns (still dark)

First build package of the audited slice-3 plan
([docs/runs-slice-3-plan.md](../runs-slice-3-plan.md) §4 WP1, decided
v0.157.7): a `worker_invoke` item's agent turn now executes as a durable DBOS
workflow in `apps/api` instead of in-process in the runs worker. The feature
stays dark behind `MANTLE_RUNS`.

## Changed

- **Worker turns are DBOS workflows** (`runsWorkerTurnWorkflow`,
  `apps/api/src/workflows/runs-worker-turn.ts`): the whole slice-2 turn body
  — `'inherit'` route resolution, the depth-2 tool loop, mechanical evidence,
  output spill, `completeItem` — moved to the durable runner, running under
  `withDurableSteps` so every LLM call and tool dispatch is a journaled step.
  A crash mid-turn resumes from the last completed step instead of re-running
  the turn wholesale or eating the 600 s deadline (this closes the "worker
  turns have zero mid-turn durability" gap from the handover §5).
  `completeItem` / `requeueForRetry` are journaled too, so a crash after
  completion replays the recorded post-commit actions instead of dropping
  them (duplicates no-op at the CAS; the sweep stays the loss backstop).
- **The runs worker's lane is claim-context only**
  (`apps/web/lib/runs/execute-worker.ts`): claim under the per-run cap
  exactly as before (the engine contract is unchanged — plan C4), then
  enqueue the workflow by name via `DBOSClient`
  (`apps/web/lib/runs/dbos-enqueue.ts`, the worker-safe sibling of the
  server-only web seam) and ack. Fire-and-forget; no cross-process await.
- **Deadline re-stamp at execution start**: RUNNER_QUEUE wait time no longer
  burns an item's execution budget — the workflow re-stamps `deadline_at`
  (state-guarded) when it actually begins; an enqueued-but-never-started
  workflow still times out on the claim's original stamp.

## Added (deferred WP1 items from the plan §8)

- **Honest enqueue failure**: DBOS system DB unreachable while pg-boss is
  healthy → the item completes `failed(dispatch_failed)` immediately with
  the true cause, instead of rotting 10 minutes into a lying `timeout`.
- **`workflowID = itemId:attempt`**: idempotent per semantic attempt (a
  retry gets a fresh workflow; a duplicate wake-up dedupes at the DBOS
  layer) and deterministically derivable from the item row — a stuck item's
  workflow is one `getRun()` away, no `listWorkflows` archaeology.
- **Flag plumbing across runtimes**: `MANTLE_RUNS` (+
  `MANTLE_RUNS_WORKER_CONCURRENCY`) now pass through the compose `app-env`
  anchor, reaching web, `worker_runs` AND `api` from the host `.env`. The
  api-side workflow refuses with a structured, counter-driving
  `failed(disabled)` when the flag is off there — flag discipline does not
  fork across runtimes.

## Notes

- Observability: the workflow tags `mantle.runner='runs_worker_turn'` +
  run/item/owner ids on its span; run records land in the same
  `apps/api/src/runs.ts` read layer as every other runner.
- Replay caveat (expected, not a bug — plan §8): trace rows + token/cost
  accumulation live outside the journal, so a crash-resume re-creates trace
  rows and under-reports replayed steps' cost on the item row. Same
  characteristic as the assistant turn.
- Typecheck green across `@mantle/runs`, `@mantle/api`, `@mantle/web`; the
  engine suite is untouched (no engine transitions changed). WP2 (resume
  turns) is next, gated by the extended crash-test per the plan's acceptance
  criterion.
