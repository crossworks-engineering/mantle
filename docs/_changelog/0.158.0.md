# v0.158.0 — `ask_human`: the run reaches you, and asks properly (still dark)

A run parked on a question was **silent and bare**. Silent: the engine
inserted its `pending_tool_calls` row inside its own transaction and nothing
fired the approval fan-out, so the live badge never lit, no companion push
went out, no Telegram card arrived — you found a blocked run by happening to
visit `/pending`. Bare: the answer surface was one text box, so a step that
needed three decisions had to ask for them in prose and hope.

Both halves close here. Still dark behind `MANTLE_RUNS`; no live server, DB or
brain was touched.

## Added — the fan-out

- **`pending_created` post-commit action.** Both silent creation sites (the
  `ask_human` promote, the `run_budget` budget pause) now return an action
  carrying the inserted row's id + args, and `enqueueRunActions` delivers it
  before it touches pg-boss — a question is still announced on a brain whose
  boss connection is sick.
- **`registerPendingCreatedNotifier` seam** (`packages/runs/src/notify.ts`),
  the `registerAgentInvoker` idiom. It exists because `@mantle/tools` already
  imports `@mantle/runs` (`applyHumanAnswer`, `RUN_BUDGET_TOOL_SLUG`), so the
  engine calling the fan-out directly would close an import cycle.
  `@mantle/tools` registers `notifyPendingCreated` at module load, which every
  process that can create a question already imports (web, `worker_runs` via
  `execute-item`, apps/api). One function call lights the sidebar badge
  **always**, and then follows `reminderChannel`: a `mobile` operator gets the
  companion push, a `telegram`/unset operator with a paired chat gets the
  Telegram card — never both.
- **Advisory by design.** Losing the notice loses a ping, never correctness —
  the row is the truth and every surface catches up on next load. So it gets
  no sweep re-send machinery, an unregistered notifier warns once, and a
  throwing notifier is logged and swallowed.
- **Fired detached, never awaited.** `enqueueRunActions` is awaited by
  `settleAskHuman` *before* it writes the settle receipt (`executed_at`), and
  the fan-out ends in a Telegram `sendMessage` whose client default timeout is
  500 s. Awaited, one unreachable Telegram call would park an answered
  question in the decided-but-unsettled window past the sweep's 3-minute
  janitor, which would revert it with "the decision was interrupted before it
  applied" — about a decision that had already applied — and would also stall
  the `run.tool` worker lane and any interactive `run_plan` call. It is `void`ed,
  matching the tool-loop's existing idiom.

## Added — the questionnaire

- **`form` on an `ask_human` leaf**: `form.questions[]`, up to 4
  sub-questions, each with `{label, description?}` options, `multi_select`,
  and an `allow_other` free-text escape. `allow_other` defaults **ON** — a
  question with no escape hatch forces a wrong answer when none of the
  options fit. Bare strings are accepted as option shorthand. Every question
  gets a stable id (explicit → slugified header → positional), because the
  answer payload references questions by id, not by an array order the
  operator can't see.
- **Caps are a contract**, not advice: 4 questions, 8 options, 24-char
  headers, 80-char labels, 8 KB total. The UIs render whatever the parser
  admits, so an unbounded form is an unanswerable screen. Each rejection
  names the fix (`packages/tools/CLAUDE.md`) — "split the rest into a later
  ask_human step", "let `allow_other` cover the rest". A question with **no
  options and `allow_other:false`** is refused outright: it would render zero
  controls and disable submit for the whole form, leaving Reject as the only
  exit.
- **A bare "yes" cannot answer a questionnaire.** Approve/reject-only
  surfaces (the Telegram card, `pending_approve` with no payload) would
  otherwise complete a 4-question form with the string `'approved'` and let
  the run proceed on none of it. Such a decision is handed back with a
  teaching error pointing at `/pending`; rejecting still works, because
  declining to answer is a real answer.
- **Structured answers** ride back as
  `answers: [{question, selected[], other?}]` through PATCH
  `/api/pending/:id` and MCP `pending_approve`, and land on the item as BOTH
  `result.answers` and `result.answer` (rendered prose) — every pre-existing
  consumer, including the resume prompt, keeps reading `result.answer`
  unchanged; the prose is now capped at 4 000 chars (matching the API cap).
  The compiled state now renders an answered question's outcome instead of
  dropping it.

## Added — where you actually are

- **`<QuestionnaireCard/>`** (`apps/web/components/pending/`) — ONE renderer
  for three shapes (form → option cards + Other; flat `options` → one-click
  chips; neither → free text), with consequence-named actions for a
  `run_budget` pause. `/pending` was refactored onto it, so the approvals
  screen and the assistant panel cannot drift apart.
- **`<PendingQuestionsStrip/>` in the assistant thread** — a blocked run is
  answerable beside the conversation, above the active-runs strip (a question
  is the only thing there you *must* act on). Self-hiding.
- **The assistant button pulses** with a count while questions wait, and
  clears when the panel opens — glancing at it is the acknowledgement, not
  answering it.
- **A sticky toast** (`<PendingQuestionWatcher/>`, headless, in the shell)
  with an "Answer" action that opens the assistant. Never on first load: the
  first SUCCESSFUL fetch seeds the seen-set, so opening the app with three
  old questions waiting produces no toasts — only questions that arrive while
  you're here. One toast per row, ever. (Seeding keys off `isSuccess`, not
  `!isPending`: react-query reports `isPending === false` for a fetch that
  ERRORED with no data, and seeding an empty set from that would toast every
  pre-existing question the moment a retry succeeded.)
- **`ToastApi.push` gained an optional `action`** — the first toast in the
  app with a follow-through.

## Changed

- `useRealtime` opens a dedicated SSE stream per call, so three new consumers
  would have burned the browser's per-host connection budget for no extra
  information. The watcher now holds the **single** app-wide
  `pending_tool_call` subscription and every consumer repaints off its
  invalidation of the shared `['pending']` query. `/pending` dropped its own
  redundant subscription — net unchanged *there* — while every OTHER route now
  carries the watcher's stream, and one shared `GET /api/pending`, that it did
  not before. Three consumers still cost one fetch (`staleTime: 30s`,
  `refetchOnWindowFocus: false`, and the shell persists across client-side
  navigation, so nothing remounts).

## Tier-2 hardening (post-audit)

Six findings from the same audit round, fixed after the tier-1 pass:

- **Answers are validated against the item's own form** — unknown question id,
  duplicate entry, a pick that isn't one of that question's options, or free
  text on an `allow_other:false` question is refused with a teaching error and
  the decision is handed BACK (nothing applied, question still open). This was
  a plan item (WP1.3) that shipped unbuilt: the answers land in
  `result.answer`, which is read into the **resume prompt**.
- **The prose is labelled by question, not by id.** `renderFormAnswers` now
  takes the form and heads each line with the header or question text — a
  resumed responder reading `q1: production` learns nothing.
- **The decision is no longer truncated at 240 chars** into the resume prompt
  (`ANSWER_MAX_CHARS` = 4 000, matching the cap already on the prose), and the
  per-question lines are joined with ` · ` before the one-line render so
  "Target: production" and "Timing: tonight" can't run together.
- **`pending_created` is replay-safe inside DBOS.** `emitDurable` journals the
  fan-out and leaves the queue jobs as bare glue — queue duplicates no-op at
  the CAS, a notice has none, so a crash-replay would otherwise re-send the
  card/push.
- **The app-wide query narrowed** to `?status=pending&limit=50` under its own
  `['pending-questions']` key (was `?limit=200`, no status filter, sharing
  `['pending']` with a different `queryFn` — the payload depended on whichever
  observer fetched first).
- **In-progress answers survive a card unmount.** The strip holds the drafts
  and orders questions oldest-first, so a newly-arrived question no longer
  shifts every card down or silently discards a half-filled form.

## Tests

`pnpm verify` green: 2516 → **2555**; the DB-backed engine suite 36 → **38**. New: the notifier seam (3, no DB — runs
in CI), action routing incl. the never-awaited guarantee (`boss.test.ts`, 6),
form parsing incl. every cap, the unanswerable-question refusal and their
teaching errors (9), the defensive client-side form parse (9, malformed input
degrades to "no form" rather than throwing mid-render), and settle coverage
for structured answers and the bare-approval refusal (6). The DB-backed
engine suite was run against the workstation Postgres over an SSH tunnel —
**37 green** (36 → 37), asserting both creation sites emit the notice with
the right row id and that a form reaches the pending row verbatim, with the
exactly-one-resume and lock-ordering invariants unchanged.

Each new guard was **falsified before being trusted** — re-introducing the
defect makes exactly its own test fail and nothing else:

| Guard | Re-introduce the defect → |
|---|---|
| the fan-out is never awaited | `boss.test.ts` "does not await the fan-out" hangs 5 s, then fails |
| a bare approval can't answer a form | `pending.test.ts` "refuses a BARE approval" fails |
| an unanswerable question is refused | `builtins-runs.test.ts` "refuses an UNANSWERABLE question" fails |
| a replay doesn't re-announce | `runs-worker-turn.test.ts` "does NOT re-announce a question on replay" fails |

## Not covered

Known gaps, stated plainly rather than left to be discovered:

- **Telegram is still approve/reject only** (plan WP4, deferred). The card
  renders a form as `form={…}`, so the sub-questions are invisible on a phone;
  a form row can now only be *rejected* from Telegram, or answered on
  `/pending`. The companion push copy is likewise still generic.
- **The WP0 emission is only asserted in the DB-gated engine suite**, which
  skips in CI; the routing split has its own `boss.test.ts`, but the
  "does an engine transaction emit the notice" assertions need
  `RUNS_TEST_DATABASE_URL`.
- **No component tests** for the new UI (this repo has no jsdom). The cards
  were verified in a browser against the workstation stack — form,
  flat-options and budget shapes, single- and multi-select, the
  disabled-until-complete rule, the settle-revert banner, live repaint and
  toast on a new question, light and dark, desktop and mobile — but that is a
  manual pass, not a regression gate.
- **The end-to-end fan-out on a live brain** is untested: dogfood territory
  (`MANTLE_RUNS=1`, grant the `runs` group, plan a run with an `ask_human`
  step carrying a `form`).
