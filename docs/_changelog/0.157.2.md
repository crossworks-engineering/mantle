# v0.157.2 — Runner queues: the spine's foundation (schema + completion engine)

First cut of the runner-queue architecture (slice 1 of 3): durable, inspectable
execution plans for delegated work. A **run** is a tree of **run items** —
structured concurrency with `group_seq` / `group_par` interior nodes and
`tool_call` / `worker_invoke` / `audit` / `ask_human` / `note` leaves. Items
are immutable once created (append-and-supersede, never edit): the queue IS
the audit log, and the responder resumes from compiled queue state rather than
held context.

Nothing calls this yet — the engine is inert groundwork. The dispatcher,
`run_*` responder tools, deadline sweep, resume turn, run view, and the
`runs.enabled` gate follow in the rest of slice 1.

## What changed

- **Migration 0129** — `runs` + `run_items`. Soft agent/turn refs (audit
  history survives agent deletion with ids intact); costs in integer micro-USD
  (the `traces` unit); subtree-cascading self-FK on `parent_id`; partial index
  for the deadline sweep. Additive only, no data change.
- **`@mantle/runs`** — the execution engine (`createRun`, `claimItem`,
  `completeItem`, `appendChildren`, `cancelRun`). Every transition is a CAS;
  side effects (pg-boss enqueues) are returned as post-commit actions, so
  at-least-once job delivery becomes effectively-once semantics and a crash
  between commit and enqueue is sweep-healable.
- **The resume-storm solution, proven**: child completion increments the parent
  group's `children_done` under its row lock; exactly one transaction observes
  `done == total`, seals the group, bubbles upward, and (at the root) emits
  the single resume. The DB-backed test battery (gated on
  `RUNS_TEST_DATABASE_URL`, self-provisioning scratch database) covers the
  parallel-completion race 2-wide × 12 rounds and 8-wide, duplicate-delivery
  no-ops, seq promotion, nested bubbling, `wait_all` / `fail_fast` join
  policies, sealed-append rejection, empty groups, and cancel-then-late-finish.
