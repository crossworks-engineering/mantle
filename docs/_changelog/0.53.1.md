# 0.53.1 — 2026-06-24

An empty cell no longer counts as "less than" a number — ordered filters stop
matching blank rows.

## Highlights

- **Empty cells never match ordered filters.** `queryRows`' `matchesFilter` fell
  back to string comparison when a cell or target wasn't numeric. For an *empty*
  cell against a numeric target that meant `'' < '30000'` === true — so
  `Design Pressure < 1000` matched every blank-pressure row, not just real ones
  below 1000. On a real circuit register that inflated the result from 225 real
  matches to 657 (the 432 pressure-less rows leaked in). The `gt/lt/gte/lte` ops
  now return `false` for an empty cell before the string fallback (SQL NULL
  semantics — a blank has no magnitude or order).

## Notes

- `eq/neq/contains` keep their string behaviour; `empty` / `notEmpty` are the way
  to ask about blanks; ordered comparison on a populated **text** column still
  uses the lexicographic path.
- Verified on the live register: `DP<1000` 657 → 225, with `empty`=432 /
  `notEmpty`=580 unchanged (225 + 432 = 657).
- Tests: table-model 36/36 (new: empty excluded from `gt/lt/gte/lte`, text
  compare intact), content + tools 476/476. No schema migration in this release.
