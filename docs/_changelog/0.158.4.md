# v0.158.4 — restore the Dockerfile's workspace-manifest invariant

`packages/runs` was added during the runner-queue work without being added to
the Dockerfile's manifest-copy list. The Dockerfile documents that list as an
invariant — it must name every workspace `package.json` — and ships the check
that proves it, which has been failing since:

```sh
diff <(grep -oE '(apps|packages)/[a-z-]+/package.json' Dockerfile | sort -u) \
     <(find apps packages -maxdepth 2 -name package.json -not -path '*/node_modules/*' | sort)
```

## Fixed

- `COPY packages/runs/package.json` restored to the deps stage, so the list is
  exhaustive again and the drift check passes.

## Why this was not breaking the build

Worth recording, because the comment above the list predicts a hard failure
("`pnpm install --frozen-lockfile` fails … the workspace it sees doesn't match
the lockfile") and that is not what happens.

`--frozen-lockfile` installs what the **lockfile** says, and `packages/runs` is
a lockfile importer. pnpm materialised its `node_modules` from the lockfile
regardless of the missing manifest — verified in the shipped v0.158.1 image,
where `packages/runs/node_modules` carries `drizzle-orm`, `pg-boss`, `postgres`
and the `@mantle` links, and `apps/web/node_modules/@mantle/runs` resolves.
v0.158.0, v0.158.1 and v0.158.3 all built green from `main` and run the runner
engine in production.

So this is hygiene, not a repair: it removes a latent fragility that happens to
be harmless under pnpm 11.1.2's frozen-lockfile behaviour, and it makes the
repo's own stated check true again. The comment's warning is left as written —
it is the right instinct even if this pnpm version is more forgiving than it
claims.
