# 0.66.0 — 2026-06-26

A foundational release with little to click but a lot underneath: your
assistant turns now run on a **dedicated, always-on backend that survives a
restart**, and the web app became a **true client/server split** — the browser
no longer touches the database directly. You won't see new buttons; you'll feel
it as turns that don't die when you navigate away and a much cleaner path to a
desktop (Electron) app and laptop-only dev.

## Highlights

- **Durable assistant turns (`apps/api` + DBOS).** Long LLM/agent work moved off
  the web request into a dedicated Node service that runs each turn as a
  **durable [DBOS](https://docs.dbos.dev/) workflow** — journaled to a Postgres
  system database, step by step. If the web process restarts (or you reload the
  page) mid-turn, the turn keeps running and finishes; the work is no longer tied
  to the request that started it. Per-step idempotency is proven by a
  crash-recovery harness (`apps/api/src/crash-test.ts`). The Telegram responder
  loop became a durable workflow too, and the old `apps/agent` service was folded
  into `apps/api` — one runner for all background agent work.

- **The web app is now a pure client.** Every screen — pages, notes, todos,
  events, contacts, tables, files, the whole `/settings/*` surface, the
  dashboard, traces, and `/assistant` — was converted from server-rendering
  against the database to **fetching its data over `/api` from the browser**
  ([TanStack Query](https://tanstack.com/query)). Server actions are gone. The
  result: **no code in the browser bundle imports the database** — the app talks
  to the backend only over HTTP, which is exactly what an Electron desktop client
  or a detached dev session needs.

- **Bearer-auth + CORS for detached clients.** Every `/api` route now answers a
  bearer token (the same mechanism the mobile companion already used) and returns
  a clean **401** instead of a login-page redirect when called without a session,
  with opt-in CORS for a separate origin. This is what lets a future desktop app —
  or a laptop with no database credentials — authenticate and run against a remote
  API.

- **Database-less local development.** Because the frontend fetches everything over
  `/api`, you can now run the full UI pointed at a **remote/deployed API with just
  a bearer token** — no local Postgres, no DB credentials on your laptop. One of
  the two original goals of the split, delivered.

## Notes

- **Shared wire types (`@mantle/client-types`).** The browser and server share a
  set of typed DTOs (agents, AI-workers, tools, tool-groups, content rows, …) so a
  drift between what the API returns and what the client expects is a **compile
  error**, not a runtime surprise — the deliberate alternative to blanket
  response-validation.
- **Signed asset tokens** let raw image/attachment `src`s resolve from a detached
  client without a session cookie; a few security hardenings rode along (rate-limit
  on the proxy-observed client IP, no `*` CORS wildcard on the auth surface).
- **Deploy:** brings up the new `api` runner service; the one-shot `migrate`
  provisions the **DBOS system database** before any app service starts. Schema:
  migration **0105** adds durable turn state (`status` + `error`) to
  `assistant_messages`. The runner (`apps/api`) must be running for the assistant
  to work — it's in `pnpm dev` and compose.
- Architecture reference: [`architecture.md` §3a](../architecture.md) (durable
  runners + the split) and the completion record in
  [`fe-be-split-session-handover.md`](../fe-be-split-session-handover.md).
