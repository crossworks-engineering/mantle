# v0.131.1 — First-time peer pairing unblocked

## Fixes

- **Two fresh Mantles can pair now.** Connecting brains was a chicken-and-egg
  deadlock: "Add peer" required the *other* side's token up front, but your own
  token is only minted *by* adding the peer — so first-time pairing had no
  legitimate path (hit live on a pilot deployment). The peer's token is now
  optional at create: add a peer with just a name and base URL and your token
  is minted and revealed immediately — the thing you actually need first. The
  peer sits in a new **awaiting-their-token** state in which inbound requests
  already verify (the other side can query you as soon as they paste your
  token) while outbound calls are cleanly disabled with a "finish pairing"
  hint. Pasting their token in the existing detail-pane field completes the
  pairing and activates the peer.

- **The two-token dance is now explained where it happens.** The create screen
  walks the two steps explicitly — Step 1: send them *your* token (shown
  once); Step 2: paste *theirs*, now or later — and pending peers carry an
  "awaiting their token" status in the list and detail views. Agent-side,
  querying a half-paired peer returns a friendly "paste their token at
  /settings/peers" error instead of a generic failure.

Migration `0117` (drops the NOT NULL on `mantle_peers.outbound_token_enc`) runs
automatically via the standard `db:migrate` on upgrade.
