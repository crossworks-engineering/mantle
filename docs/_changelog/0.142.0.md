# v0.142.0 — The Team Forum, and a brain that knows what's current

Team conversation moves from private 1:1 chats to a **shared forum**: topics
every member can read, answers written for the room, and an agent that finally
sees the whole thread. Underneath it, two kinds of staleness die in one
release: retrieval learns which content has been **superseded** by newer
copies, and the deployment stack learns to keep its own **shape** current so
an update can never silently miss a piece of the stack again.

## The Team Forum

- **Shared topics replace the 1:1 assistant chat.** Members start topics
  (question, review, feature, bug) and post into them; every team member reads
  the thread, and user posts are name-prefixed so the agent — and everyone
  else — can track who said what. The agent answers into the topic, addressed
  to the asker but written for the room. The old 1:1 chat lives on at
  `/team/assistant` as a **read-only archive** — the write path returns 410
  (set `TEAM_CHAT_POST_ENABLED=1` to reopen it deliberately).
- **One agent turn per topic at a time — with the full thread in view.** Turns
  in a topic are serialized by a partitioned queue (different topics run in
  parallel), and each turn loads the transcript at the moment it actually
  runs — so a queued answer always sees every prior answer, and two members
  posting at once can never talk the agent into contradicting itself.
- **Private topics.** A member can mark a topic private — visible to its
  author and you, invisible to everyone else, enforced by a single tested
  visibility rule shared by every query and write path.
- **Owner controls.** Team admin grows a **Topics** tab: pin/unpin, reply as
  yourself, and **mark a post as the answer**. The Chats tab's recent-access
  inspector is now resizable and collapsible.
- **One budget, no loopholes.** Chat and forum spend count against a single
  daily cap per member (`TEAM_CHAT_DAILY_TURNS`), whichever surface is used.
  Raw agent error strings no longer reach members — a failed turn shows a
  generic notice; the detail stays owner-side in traces.

## Content that knows it's been replaced

Old copies stop outranking their corrections. Nodes gain a supersession
lineage (`superseded_by`, mirroring what facts have had for months), and
retrieval uses it everywhere:

- **Automatic lineage.** Importing a file or promoting notes into a page marks
  the source superseded by the new page (`supersede_source: false` opts out).
  Versioned export families (`…_v01` / weekly dumps) mark older siblings
  superseded by the newest, and never touch a mark you set by hand.
- **Corrections in one sentence.** Tell your assistant "this file is outdated —
  the page is the current version" and the new `content_supersede` tool stamps
  it (new **Content curation** tool group; owner-side only — members can't
  re-weight your brain). Fully reversible; nothing is ever deleted.
- **Retrieval prefers the living copy.** Superseded content is down-weighted
  (still findable — a nudge, never a filter), and when an old copy still
  surfaces, the hit is flagged **SUPERSEDED** with a pointer to the current
  version — chains resolve to the newest end — so the assistant cites the
  replacement instead of quoting the corpse.

## Outlook mail, complete

- **Send via Microsoft Graph.** Outbound mail from a connected Microsoft
  account (Mail.Send scope, inline attachments) — the assistant can now reply
  from your Outlook address, not just read it. Reconnect nudge in settings
  when the scope isn't granted yet.
- Backfill and sender-discovery now cover Microsoft accounts (including
  `@domain` wildcards), and the account-settings copy no longer pretends
  every provider wants an IMAP password.

## The stack keeps its own shape current

The recurring failure where a routine update ran the new image on an old
`docker-compose.yml` — silently missing a new service or mount — is closed:

- **The release owns the compose file.** The image embeds its canonical
  `docker-compose.yml`; on update, a box whose compose is untouched since the
  last release gets the new one swapped in **in the same roll** (previous file
  kept as `docker-compose.yml.prev`). A hand-edited compose is never
  overwritten — the update proceeds and the drift is reported loudly instead.
- **Box-local customization has a home**: `docker-compose.override.yml` (+
  `.env`) — Compose merges it natively, so your changes survive every refresh.
  `scripts/compose-adopt.sh` migrates an existing box once: it diffs your
  compose against the canonical, helps port edits to the override, and
  `--apply` installs the canonical + baseline.
- **Drift is visible even when it can't be fixed.** `/settings/updates` now
  shows the stack state — in-sync, stale, modified, or no-baseline — computed
  from the image's embedded canonical vs. the actually-running stack. "Image
  is release X, compose is release Y" is no longer a silent condition.
- **The watcher is watched.** The `autoheal` sidecar (added in v0.141.0) gets
  a real healthcheck — a Docker-socket ping replacing the image's tautological
  process check — so a wedged autoheal shows `unhealthy` instead of looking
  alive while healing nothing.
