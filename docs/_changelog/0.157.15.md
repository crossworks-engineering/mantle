# v0.157.15 — Runner queues slice 4: operational readiness (still dark)

Slice 4 turns the audited slice-3 engine into an operable surface: a
dedicated background queue, the worker template in the manifest, real
answering + management UIs, and a first-class run view. The feature stays
dark behind `MANTLE_RUNS` — none of this is reachable until a brain sets the
flag and grants the `runs` tool group.

## Added

- **Dedicated runs-turn queue (WP-F).** Background runs turns (worker +
  resume) previously shared `RUNNER_QUEUE`'s FIFO with the owner's
  interactive assistant/telegram turns, so a run fanning out N worker turns
  could queue ahead of a live chat message — the slice-3 starvation
  watch-item. New `RUNS_TURN_QUEUE` (`'mantle.runs'`, `packages/runs`)
  registered in apps/api with its own cap (`runsTurnConcurrency()`, env
  `MANTLE_RUNS_TURN_CONCURRENCY`, default 2); both enqueues in
  `apps/web/lib/runs/dbos-enqueue.ts` now target it. Same isolation
  `FORUM_QUEUE` gives topic turns. Deploy-skew note: a worker enqueuing
  before apps/api restarts with the queue registered leaves jobs waiting
  until the api rolls — compose restarts both together, so it is transient.
- **Default worker agent in the system manifest (WP-E).** The runner-queue
  worker template ships as a `MANIFEST_AGENTS` entry (role `worker`, model
  the `inherit` sentinel, tool group `memory-core`), importing its
  slug/prompt/groups/model constants from `@mantle/runs` so the template
  stays single-sourced. Onboarding + the boot reconcile now seed the row on
  every brain; `ensureWorkerAgent` stays the lazy fallback (finds the seeded
  row — no duplicate). Still harmless while dark: the `runs` group is
  unattached to the persona, so a dormant worker template does nothing until
  a run invokes it.
- **`/runs` — a first-class master-detail run surface (WP-B).** Promoted
  from the `/debug/runs` tab to a top-level screen (nav System group): left
  = the owner's runs (title, status incl. `paused`, created, cost;
  URL-driven selection + pagination); right = the compiled item tree with
  the operator Cancel actuator (AlertDialog) plus "needs you" banners
  linking to `/pending` on a budget pause or an open question. The API
  moved `/api/debug/runs/*` → `/api/runs/*` (handlers unchanged, owner-scoped);
  the list route gained pagination and an `?active=1` filter.
- **`/settings/worker-groups` (WP-D).** A master-detail screen over the
  `agent_groups` panels that were previously MCP-only: left = groups (name,
  slug, member count, enabled); right = detail form (name, enabled Switch,
  members as a checkbox list of enabled worker agents, ghost Delete); create
  via a slug+name Dialog. Owner-scoped API under
  `/api/settings/worker-groups`; the membership validation (enabled worker
  slugs, 1..10, slug ≤64) mirrors the `worker_group_ensure` MCP tool
  (shared helper `apps/web/lib/worker-groups.ts`).
- **Active-runs strip on `/assistant` (WP-A).** A self-contained
  `ActiveRunsStrip` at the top of the assistant thread: a compact card per
  running/paused run (title, leaf progress, spend vs budget, `paused` /
  `waiting` badges, Cancel, Open → `/runs`). Polls `GET /api/runs?active=1`
  every 5s only while a run is active; renders nothing when there are none
  (zero cost on brains that don't use runs). No new realtime channel —
  polling only.

## Changed

- **`/pending` answers `ask_human` / `run_budget` (WP-C).** The approvals UI
  is now slug-aware: an `ask_human` row shows the question as its headline
  with one-click `options` chips and a free-text "Answer & approve" field
  (PATCH `/api/pending/:id` `{decision:'approve', answer}`); a `run_budget`
  pause labels its actions "Raise budget" / "Cancel run". A still-pending row
  carrying a non-null error (the v0.157.14 settle-revert path — a bounced
  decision) surfaces a loud "decide again" banner.

## Notes

- **Still dark.** No `MANTLE_RUNS` change; no live server, DB, or dev-brain
  touched. Nav visibility: the sidebar is a static list with no per-item
  conditional-visibility mechanism (only the pending badge is dynamic), so
  the new "Runs" and "Worker groups" items ship always-visible like their
  siblings rather than gated on `isRunsEnabled()`/run count.
- **Deferred / skipped.** The optional inline "run started" chat card (the
  assistant turn stream doesn't expose completed tool-call results
  client-side — it would need stream/protocol changes); WP6 (partitioned
  per-run cap) remains deferred from slice 3.
