# 0.160.0 — 2026-07-22

**Formulas** — a new content type for calculations taken from published
standards, plus a real maths engine underneath the whole app.

A calculation lifted from an engineering standard is not one formula. It is
scalar expressions, a branch on a threshold, keyed lookup tables, and prose
rubrics that map a described system to a rating — and only the first of those is
arithmetic. A format that stores expressions cannot hold the model, which is why
this exists.

## Highlights

- **`formula` node type** — a declarative spec (variables, equations, piecewise
  branches, lookup tables, classification criteria) stored in `nodes.data`, with
  no sidecar. Summarised, embedded and searchable like everything else;
  `search_chunks` covers it. Migration `0136` adds the enum value.
- **`/formulas`** — master-detail viewer. Renders equations through KaTeX,
  the variable table with units and roles, every lookup as a table, and the
  rating criteria. An evaluator panel computes any target and shows the
  **derivation**: which branch fired, which row matched, what each symbol
  resolved to.
- **Seven new tools** — `formula_list` / `_get` / `_create` / `_update` /
  `_evaluate` / `_delete`, plus **`calculate`**, a general unit-aware calculator
  granted to the persona. There was no maths tool in the registry before, so an
  agent asked a bare arithmetic question answered from its head — the one thing
  models are reliably bad at, and the failure is silent.
- **mathjs replaces the hand-written formula parser** for table formula columns.
  Pinned exactly at 15.2.0 — an engineering number has to reproduce
  byte-for-byte years later, which outweighs free upgrades.
- **Dimensional analysis.** `unit` stops being display text and becomes a
  constraint: each expression is evaluated with unit-bearing quantities and its
  result dimension checked against the declared one. This catches the error no
  proofreading reliably finds — a term dropped inside a `SQRT`, or a constant
  labelled with the wrong dimension.

## Behaviour change — table formula columns

`+` no longer concatenates strings; use `CONCAT`. Excel requires `&` and
Airtable requires `CONCATENATE` for the same reason: making `+` loose enough to
join text means loosening the type discipline that makes unit arithmetic work.
An early cut of the migration did exactly that and silently broke `1 ft + 2 ft`.

Blank and unknown cell references still read as `0`, applied when binding the
scope rather than in the type system, so spreadsheet ergonomics cost nothing at
the unit layer. Everything else is unchanged — a differential test runs every
expression through both engines and fails on any undeclared divergence.

`table-formula.ts` is retained for one release as a revertible fallback and as
that test's baseline. Nothing else imports it.

## Notes

- No deployment ceremony. Migration `0136` is a single additive
  `ALTER TYPE … ADD VALUE`, in its own file because that statement cannot share a
  transaction with anything referencing the new value.
- `/tables` first-load JS grows 250 kB → 428 kB, the cost of putting a real
  maths engine in the browser where `resolveCell` runs.
- Evaluation **fails loud**, unlike formula columns: a missing or misspelled
  input is an error, never a zero, and symbols are case-sensitive because `k`
  and `K` are different quantities in the vapour equations.
- Lookup tables are stored as data rows, never as a nested `IF()` chain — so a
  revised factor is a one-line diff, and `checkLookupCoverage` can prove a table
  is total over its declared keys instead of hoping.
