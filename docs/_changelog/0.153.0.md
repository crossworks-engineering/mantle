# v0.153.0 — Nightly maintenance cron + unified run history

Phase 2 of the maintenance-runner plan (docs/maintenance-runner.md) — the last
of the three phases.

## What's new

- **`worker_maintenance`** — a dedicated pg-boss cron worker
  (`apps/web/workers/maintenance.ts`, queue `mantle.maintenance.sweep`,
  `'30 3 * * *'` nightly) that runs the registry's `schedulable` sweeps
  **in-process**: today exactly one, `entities-dedupe` auto tier (pure SQL,
  no model calls). The tier-merge logic was lifted into a shared
  `runEntitiesDedupe()` (`lib/maintenance/sweeps.ts`) used by both the CLI
  script and the cron — one definition of the hygiene job. New compose
  service (autoheal + healthcheck) and a `maint` lane in root `pnpm dev`.
- **`maintenance_runs` table** (migration 0128) — unified run history: every
  invocation from the CLI (`pnpm maintain`, best-effort), the Maintenance
  tab, or the cron records slug, source, live/dry-run, state, timings, exit
  code, and a one-line summary. The Maintenance tab now shows the last 20 as
  a **History** table.
- **Double-fire guard** — the cron skips a sweep with any `cron`-sourced run
  in the last ~20h (failures arm the guard too, mirroring the backups
  scheduler), on top of pg-boss's once-per-slot semantics.
- Cost-safety unchanged and re-asserted: the cron path re-checks
  `cost === 'sql'` before running anything; model-spending tasks cannot be
  scheduled (registry assertion + tests).

## Deploy

- Migration 0128 runs via the normal `migrate` gate.
- **New compose service `worker_maintenance`** — `docker compose up -d` picks
  it up from the updated compose file; no volumes, no new env.
