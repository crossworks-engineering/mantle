# v0.153.2 — Stop actually stops tool-heavy turns

Fixes the "Stop doesn't stop it" bug in the assistant chat: the Stop button
cancelled the model's **text generation** (the original live-turn-streaming
wiring) but the **tool loop never checked the abort signal** — so every tool
call the model had already emitted still executed (research, page edits,
`invoke_agent` delegation…), the loop kept iterating against a dead signal,
and could even fire the "your previous response was empty, reply now" retry
after the user asked it to stop. For turns that are mostly tool work — most
interesting turns — Stop appeared to do nothing.

## What changed

`runToolLoop` (`packages/agent-runtime/src/tool-loop.ts`) now honours the
turn's abort signal at three points:

- **After each chat round** — complete tool calls carried by an aborted
  round's partial reply are discarded; the turn finalizes immediately with
  the partial text (the existing Stop contract: partial kept, row
  `complete`, prompt restored to the composer).
- **Before each tool call in a batch** — nothing new starts after a Stop;
  remaining calls get a paired synthetic `cancelled_by_user` result so the
  provider transcript stays valid. The tool already in flight completes
  (tools aren't individually cancellable yet).
- **After the batch** — no further rounds; the empty-reply retry is skipped
  when the emptiness was caused by the abort.

Delegated sub-agents run the same loop with the inherited turn id, so a Stop
now halts the whole delegation chain at the next tool boundary. Net effect:
Stop lands within one tool call instead of "within one token, or never."

Covered by a new unit suite (`tool-loop.abort.test.ts`, 5 tests: discard,
mid-batch skip with pairing, post-batch halt, no empty-retry, and a no-Stop
control); the existing 58 tool-loop tests pass unchanged.
