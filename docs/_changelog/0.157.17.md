# 0.157.17 — runs: automated coverage for the LLM-dependent half

The slice-3/4 scorecard capped test coverage because "both workflows live,
telegram delivery, pending UI round-trip" had none. Most of that gap was never
really LLM-dependent — the model sits behind one mockable call, and this repo
already had the idiom (`vi.hoisted` + module mocks, as in
`responder-loop.test.ts`). What was missing was a way to test the DURABLE
behaviour. That is what landed here. **No production behaviour changed**: the
only non-test edits are two `export` keywords so the workflow bodies can be
driven directly.

## Added — the replay harness

`apps/api/src/workflows/workflow-test-kit.ts` (test-only) provides a
`DBOS.runStep` stand-in with real journal semantics: pass 1 executes step
bodies and records their results; calling the same workflow again with the
same journal serves those results back instead of re-running them — exactly
what DBOS recovery does after a crash. `crashAfter` stops a pass the instant a
named step commits.

That turns "does this survive a crash?" into an ordinary unit assertion, with
no DBOS, no Postgres, no LLM and no hand-timed `kill -9`. Both bugs the
slice-3 final audit found by hand (F1, F4) are now guarded automatically, and
**each guard was falsified before being trusted** — re-introducing the pre-fix
code makes exactly the replay tests fail, and nothing else:

| Guard | Re-introduce the defect → |
|---|---|
| resume: crash after the claim still reports, once | the 2 replay tests fail (`duplicate` instead of `reported`) |
| worker: crash past completion re-emits recorded actions | 1 test fails (`stale` instead of `done`) |
| pending: a failed settle hands the decision back | 1 test fails (the throw escapes) |

## Added — three suites (39 tests)

- **`runs-resume-turn.test.ts`** (14): pre-claim refusals (flag off, duplicate,
  budget-paused, missing item, no chat-capable agent); the v0.157.5 ordering
  (assembly → claim → loop) asserted as an observed sequence, not by reading
  the source; claim-race loss; audit mode posting nothing; telegram delivery
  plus both fallback paths (unpaired chat, disabled account) proving the report
  is never lost; and the two crash-replay guards. **apps/api's first tests.**
- **`runs-worker-turn.test.ts`** (13): stale wake-ups, `failed(disabled)` when
  the flag is off in this runtime, the deadline re-stamp, the mechanical
  evidence ledger including failed tool calls, route inheritance and its
  failure modes, retry-then-fail on empty output, transport errors as semantic
  retries, and the F4 action-replay guard.
- **`packages/tools/src/pending.test.ts`** (12): the settle layer that had no
  coverage at all — routing before tool resolution, the owner hand-off (F2),
  moved-on expiry, missing item refs, and the F3 revert-to-pending recovery for
  both `ask_human` and `run_budget`.

## Still not covered (deliberately)

Judgment QUALITY — whether a review actually catches a fabricated verification
claim — is not a unit test; it needs an eval with a gold set, modelled on
`eval:recall`. Real provider tool-calling, real Telegram acceptance, real DBOS
recovery in a container (`crash-test.ts` covers the mechanism), and browser
rendering remain smoke-test and dogfood territory.
