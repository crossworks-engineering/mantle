# 0.52.0 — 2026-06-24

Document retrieval now works in larger, more coherent passages — a procedure step or table row stays whole instead of being split across tiny fragments.

## Highlights

- **Bigger chunks.** `chunkDocText`'s default chunk size rose from ~1,500 to
  ~2,750 chars (overlap 150 → 300). A passage now holds a complete procedure
  step / table row rather than a fragment, which is what the responder reasons
  over for technical lookups.

- **Lower `chunk_limit` to match.** The default dropped from 12 to 8 (runtime
  `CHUNK_LIMIT_DEFAULT` + the manifest persona). The budget that matters is
  `chunk_limit × chunk size`: 8 × ~2,750 ≈ 22k chars — about the same context
  as the old 12 × 1,500, but far less fragmented. Raising chunk size without
  lowering the count would have silently ~doubled the budget.

## Notes

- **Takes effect for newly-extracted content immediately; existing documents
  keep their old chunks until re-extracted.** To apply the new chunking to an
  existing corpus, clear the per-node extract markers and re-run the extractor
  (it deletes + re-inserts each node's chunks in one transaction). Best paired
  with an embedder change so the corpus is re-embedded in the same pass.
- A per-agent `memory_config.chunk_limit` still overrides the default.
- No schema migration in this release.
