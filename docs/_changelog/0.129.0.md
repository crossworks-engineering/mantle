# v0.129.0 — PDF export gets its own engine

The Pages → PDF export (new in v0.128.0) no longer embeds a browser in the
app. It now runs as a **browser sidecar** — a stateless sibling service, the
same pattern as Tika — ready for team-wide use.

## Same PDF, better home

Output is unchanged: the sidecar's Chromium loads the same owner-authed print
surface with the live page CSS, so the PDF still looks exactly like the
on-screen page. What changed is where that browser lives:

- **Own container** (`browser`, browserless/chromium) with its own memory
  space — a huge or pathological page render can never stall or OOM the web
  server.
- **Session queue + hard timeout** — concurrent downloads queue in the
  sidecar (`CONCURRENT`/`QUEUED`), and a stuck render is killed at 60s.
- **No published ports in prod** — only the internal docker network reaches
  it; dev publishes `127.0.0.1:9222` for the native-node dev servers.
- **Graceful degradation** — if the sidecar is down, PDF download returns a
  clear 503; Markdown and Word are unaffected.

## A leaner app image

The app image drops the embedded Chromium + fonts (~1 GB): `puppeteer` is
replaced by `puppeteer-core` (pure driver, downloads nothing) connecting over
websocket (`BROWSER_WS_ENDPOINT`). Browser security patches now arrive by
pulling the sidecar image, not by rebuilding Mantle.

## Upgrading

`docker compose pull && docker compose up -d --wait` brings up the new
`browser` service automatically. Optional env: `BROWSER_TOKEN` (internal
auth token; defaulted), `MANTLE_PRINT_ORIGIN` (preset to `http://web:3000`).
Dev stacks: `pnpm start` (or `docker compose -f docker-compose.dev.yml up -d`)
starts the sidecar; `.env.local` needs
`BROWSER_WS_ENDPOINT=ws://127.0.0.1:9222?token=mantle` (see `.env.example`).
