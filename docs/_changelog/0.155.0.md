# v0.155.0 — App builder: `app_db_seed` reference-data loading + honest build failures

Driven by a real field failure: an owner asked their assistant to rebuild an
Excel calculator as a mini app. The calculator itself built fine, but loading
its ~2,800-row reference table into the app's SQLite hit a wall — the
recipe-tool framework has no transform primitive, the coder sandbox has no DB
access, and routing thousands of single-row `host.db.exec` calls through the
iframe bridge is not a loading strategy. The delegation chain (appsmith →
toolsmith → coder) burned four attempts and stalled; the app shipped with a
"paste your own JSON" fallback. This release closes that gap.

## What's new

- **`app_db_seed(id, table, rows, replace?)`** — a new authoring builtin that
  bulk-inserts row objects into ONE table of an app's own SQLite database,
  atomically. Row keys are validated against the live table columns (teaching
  errors name the offending row/column and the available columns); values are
  string/number/boolean/null (booleans stored as 1/0); any bad row rolls the
  whole batch back. ≤2000 rows per call — batch bigger sets with
  `replace: true` on the first batch only. Rides the `apps` tool group, so
  every agent that can author apps can now seed them. Core logic is
  `seedRowsIntoHandle` in the app broker, unit-tested against in-memory
  SQLite (validation, replace-then-append batching, atomic rollback,
  identifier/`sqlite_*` refusal).
- **`app_build` failures now fail the call** — a red compile used to return
  top-level `ok: true` with `build_ok: false` nested in the output; an agent
  scanning the conventional `ok` field read failure as success and sailed on
  to `app_publish`, hitting `NoGreenBuildError` confused. A failed build now
  returns `ok: false` with every esbuild error's file:line:column in the
  error text plus the recovery move. Success shape is unchanged.
- **Skill + guide: data-tier guidance** — the `app_authoring` skill and
  `docs/app-authoring-guide.md` now open the data section with the tiers:
  (1) pure code — calculators/converters need NO tools and NO database;
  (2) fixed reference data — seed once with `app_db_seed`; (3) live data —
  a declared api_tool. Previously the guidance was dominated by tool-wiring
  prose, which pushed agents toward toolsmith delegation for apps that
  needed none of it. Also states explicitly that a green build only proves
  the code compiles (there is no runtime-error feedback from the iframe) and
  that one-time data loads are never toolsmith work.

## Deploy

- Tag-only — no migration, no compose change. The new builtin joins the
  `apps` tool group via the boot reconcile (`seedToolCapabilities` membership
  overwrite + `applyManifest` skill-body sync), so existing brains pick it up
  on the version bump.
