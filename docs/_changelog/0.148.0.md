# v0.148.0 — Search over HTTP: the brain's search engine, opened to the companion

The hybrid search engine that agents have always had — full-text + vector,
reciprocal-rank fused, salience-aware — gains an owner-facing HTTP endpoint:
**`GET /api/search`**. Until now it was reachable only through the MCP tools
(`search_nodes` / `search_chunks`), which meant the mobile companion could not
offer a search screen without routing every query through a chat turn. Now it
can: same engine, same ranking, one authenticated GET away.

## What changed

- **`GET /api/search`** (owner-gated; mobile bearer works unchanged). Params:
  `q` (required), `mode=nodes|chunks` (default `nodes`), `type` (node-type
  filter, same enum as the `search_nodes` tool), `branch` (ltree prefix,
  validated), `tags` (comma-separated, capped at 10), `limit` (1–50,
  default 20).
- **`nodes` mode** mirrors `search_nodes`: the query is embedded
  (vector-led hybrid); a failed embed degrades to FTS instead of erroring.
  Results carry `id, type, title, path, tags, summary, updatedAt, url`, plus
  a `supersededBy {id, title, url}` annotation so a stale hit points at its
  living successor.
- **`chunks` mode** mirrors `search_chunks`: passage-level hits
  (`nodeId, nodeTitle, nodeType, ordinal, heading, text, url`). Passage
  search is vector-first, so an embed failure here is an honest `503` rather
  than silently degraded results.
- Param parsing lives in `apps/web/lib/search-query.ts` (pure, unit-tested):
  ltree-safe `branch` validation, tool-enum `type` validation, limit
  clamping.
- `docs/mobile-companion-backend.md` gains the companion-facing contract
  section.

**Deploy: tag-only bump — no migration, no compose change.**
