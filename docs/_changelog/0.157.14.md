# 0.157.14 — runs slice 3: final-audit fixes (replay determinism, approval hardening)

The independent final audit of slice 3 (2026-07-21, pre-dogfood gate)
REFUTED v0.157.12's "resume-loss gap closed" claim and surfaced two
hardening gaps in the approval paths. All three fixes land here; the audit
record is appended to [runs-slice-3-plan.md](../runs-slice-3-plan.md) §8 and
[runs-slice3-build-handover.md](../runs-slice3-build-handover.md) §2.

## Fixed

- **F1 — resume-turn replay determinism (the refuted closure claim).** DBOS
  recovery re-runs a workflow function from the top; journaled steps return
  recorded results but plain glue re-executes against CURRENT db state. The
  resume workflow's non-journaled "cheap duplicate check" read `resumed_at`
  — set by the workflow's OWN journaled claim — so every post-claim
  crash-recovery exited `'duplicate'` and silently lost the wake-up: the
  exact handover-§5 gap WP2 existed to close, reproduced empirically against
  scratch DBs. The v0.157.12 gate missed it because its harness omitted the
  pre-claim glue and its kill point (post-outbound) was the already-benign
  window. Now: every pre-claim decision (duplicate, paused, agent/key/
  adapter) lives in ONE journaled `resume_preflight` step, so a replay walks
  the identical path back to its claim. The paused refusal (WP4 amendment 4)
  lives only in preflight: a fresh turn on a paused run still refuses
  pre-claim, while a recovery that already claimed completes (mostly journal
  reads — stranding the burned claim would be the greater evil).
- **F4 — worker-turn action replay.** Same hazard class: the initial stale
  check was glue, so a crash after the journaled `complete_item` replayed
  into an early `'stale'` exit and the recorded `PostCommitAction[]` were
  never re-enqueued (sweep-healed, but the header's "replays the recorded
  actions" contract was false). The item load is now a journaled
  `load_item` snapshot; replays reach `complete_item` and re-run
  `enqueueRunActionsSafe` (duplicates no-op at the engine CAS).
- **Gate tightened + PASSED at both kill points.** `crash-test.ts`'s resume
  shape now mirrors the real workflow (journaled preflight reading the
  claim marker → journaled claim → glue → journaled outbound) and takes
  `CRASH_POINT=post_claim` (the LOSS window: the report must still arrive,
  exactly once) alongside the default `post_outbound` (the double-post
  window). Both PASS against isolated scratch DBs; the basic shape still
  passes. The plan-§8 gate wording was the root cause — it certified
  no-double-post, never no-loss.

## Hardened

- **F2 — approval-path owner scoping + reserved slugs.**
  `applyHumanAnswer` / `applyBudgetDecision` now REQUIRE an `ownerId` and
  verify the target item/run's run row belongs to it (`'forbidden'`
  otherwise) — previously the pending row was owner-scoped but its ARGS
  were trusted, so a custom tool squatting the unregistered `ask_human` /
  `run_budget` slugs could mint rows whose `item_id`/`run_id` reached
  another owner's run under an innocent-looking approve/reject click
  (reject would `cancelRun` an arbitrary run). Second layer: those two
  slugs are now RESERVED at `api_tool_create` / `recipe_tool_create` —
  engine-created rows are the only source.
- **F3 — decided-but-unsettled approvals recover.** approve/reject flips
  the row BEFORE the run-side settle applies; a crash or db error in that
  window stranded an operator-visible "approved" that never took effect,
  with no retry path (`pending_approve` refuses decided rows) — for
  `run_budget` the run sat paused forever with nothing left to approve. Now
  a settle ERROR reverts the row to `'pending'` with the cause recorded
  (decide again), and NEW sweep duty 4c does the same for the crash window
  (decided runner rows with `executed_at` NULL after a 3-minute grace).
  Double-application stays impossible: a re-decision of an already-applied
  settle no-ops at the engine CAS and expires the row.
