# v0.157.9 — Runner queues slice 3 WP2: durable resume turns (still dark)

Second build package of the audited slice-3 plan
([docs/runs-slice-3-plan.md](../runs-slice-3-plan.md) §4 WP2, amended per the
C3 verdict): the responder resume turn — root report AND audit verdict —
executes as a durable DBOS workflow in `apps/api`. The feature stays dark
behind `MANTLE_RUNS`.

## Changed

- **Resume turns are DBOS workflows** (`runsResumeTurnWorkflow`,
  `apps/api/src/workflows/runs-resume-turn.ts`): the slice-1/2 resume body
  moved from `apps/web/lib/runs/resume.ts` (deleted) into the durable runner,
  under `withDurableSteps`, with the two boundaries the audit found missing
  added as explicit journaled steps:
  - **`claim_resume`** — placed AFTER the fallible preconditions (agent, key
    decrypt, adapter, assembly) and BEFORE the LLM loop, preserving the
    v0.157.5 ordering: a precondition failure returns without claiming, so
    the sweep re-sends. Past the claim, a crash now RESUMES the turn from its
    last journaled step instead of losing the wake-up.
  - **`record_outbound`** — the root report's `recordTurn` is journaled, so a
    crash between recording and workflow completion replays without
    double-posting to the conversation.
- **The pg-boss resume lane is a relay**: the handler peeks `resumed_at` and
  enqueues the workflow on the shared RUNNER_QUEUE — its old `batchSize: 1`
  no longer serializes LLM work globally (the 30-minute audit-verdict
  deadline is no longer hostage to one slow resume); RUNNER_QUEUE's
  concurrency is the backpressure cap.
- **Dedup instead of a fixed workflowID**: resumes enqueue with
  `deduplicationID = groupId` (at most one QUEUED resume per group, clears
  when it starts). Deliberately no fixed workflowID — a workflow that errors
  WITHOUT claiming must stay rescuable by the sweep's re-send, and a fixed id
  would dedupe the rescue into a no-op against the terminal row. The
  `resumed_at` CAS remains the at-most-once turn gate.

## Notes

- **The handover-§5 resume-loss gap is NOT yet claimed closed** — per the
  plan's acceptance gate (§8 amendment 2), that claim waits for the
  crash-test against this workflow on the dev box: kill between the journaled
  `record_outbound` and workflow completion; exactly one outbound row after
  recovery. Until it passes, treat WP2 as concurrency + code unification with
  much-narrowed loss windows.
- Deliberately not journaled: the decrypted api key (no plaintext secrets in
  the DBOS system DB) — a replay re-resolves it as glue, leaving a narrow
  post-claim + key-outage loss window, accepted and documented in the
  workflow header.
- Riding-along WP2 items from the plan (channel-routed resume delivery,
  Stop→`run_cancel`, chat run card) are still open — they land as their own
  package.
- Typecheck + lint + 2445 unit tests green; the engine suite is untouched
  (no engine transitions changed).
